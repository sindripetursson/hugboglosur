Continious Integration

Continious integration server er alveg clean, þannig sjáum við skýrar dependency. Integration server er server eða virtual umhverfi sem er er sett upp í þeim eina tilgangi að vinna í ákveðnu umhverfi.

Version Control = Revision Control = Source Control

Kerfi voru með lása, minna um það í dag. Hægt að track'a breytingar aftur í tímann.

Trunk

  • The unique line of development that is not a branch(sometimes also called a Baseline or mainline)
  • Sonja segir a' þetta sé aðaldraslið, what gonna do with all that junk, all that junk inside your trunk

Branch

  • Þá er unnið á annari útgáfu af dótinu, hægt að sameina seinna.
  • Semí sandbox

Tag

  • Er svona snapshot eða eð. Getur séð hver staðan var á skrám á einhverjum tímapunkti
  • Tagg er merkt eða skýrð sérstaklega.

Allskonar annað

  • Head - nýjasta commitið
  • Revision - er ein keðja af breytingum
  • Merge - oftast eru ekki breytingar á sama stað. Conflict eru ekki hægt að automate'a
  • Revert - Lagar manns eigið sandbox aftur að þeim punkti sem seinasta uppfærlsa var gerð
  • Rollback - Hættir við commit eða bakkar um commit.. veit ekki hvort

Key advantages

  • Allir eru með sitt sandbox
  • Þetta virkar off the line
  • Hratt
  • Smíðað til að deila breitingum
  • Branching og Merging er þægilegt
  • Less management

Disadvantage

  • Það er í rauninni aldrei til seinasta útgáfa

Eininga prófanir

  • Prófa eins lítinn hluta í einu af kerfinu eins og hægt er, virkni í klasa vanalega
  • Er talin hluti af source code
  • Keyrður sem hluti af build process
  • Keyrir hratt
  • Á ábyrgð forritara

EF UNIT TEST TEKST EKKI ER KÓÐINN EKKI GEFINN ÚT

Kóði sem er einingarpórfaður hefur sín eigin test og sýnir enn fremur ætlanir forritarans með hönnunina. Stuðlar oft að loosely coupled kóða. Gott að hafa löng nöfn á testunum til að sjá hvað testin eru að prófa.

Það er erfitt að testa tightly coupled kóða

Loosely coupled kóði

Breyting í einu module krefur ekki breytinga í einhverju öðru module.

Test Driven Development

  • Snýst ekki bara um að prófa
  • Tól til að developa betri kóða
  • Rannsóknir sýna að TDD minkar defecta til muna

TDD Mantran

Design - Test - Implement - Test

Red - Green - Refactor - Repeat Þar sem:

  • Red = write test
  • Green = Write code
  • Refactor
  • Repeat

Afhverju Build process?

  • Svipað og ýta á takka bara til að build'a, testa og keyra kóða
  • Kemur í veg fyrir langar endurtekningar
  • Sem product owner að þá er teamið mitt að delivera value oftar
  • Testerinn hefur stöðugt nýjan kóða til að prófa.

Eina afsökunin fyrir að kóðu virkar ekki, sem er oftast sönn er: Þetta keyrir á vélinni minni

Build vél gengur út á það að verkefni keyri altaf á henni.

Build integration gæji tekur af source control og keyrir build scripturnar og sér nokkup verkefnið keyrist.

Build Scriptur

  • Compile'a source kóða
  • Integrate Database
  • Keyra test
  • Keyra Inspection(automated code review)
  • Deploy'a hugbúnaðinn
  • annað...

Hæðistjórnun - Dependency Management

  • Verður vandamál í stærri verkefnum
  • Gera þetta manualy verður svakalega þrytt ef að við erum með 100 libs
  • Libs geta dependað á önnur libs, en í mismunandi versionum
  • Dependency manager er staða fyrir einn mann
  • Bergþór bjó til Dependency Facism
  • Dependency management = confiugration management

Mikilvægt að að geyma integration database í spource control.

Run Inspection minkar flæknistig kóða, finnur duplicate kóða, hannar review.

Deply Software býr til artifakta( exe, dll, jar, etc), installar artifacta, restartar server. Gerir manni kleyft að vinna á deployable hugbúnaði á hvaða tímapunkti. Þegar build hefur keyrt verður að stage'a skrárnar(veit ekki hvað það þýðir), build ferlið deployar svo sjálfkrafa eða installar skrár í viðeigandi umhverfi.
