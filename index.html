Continious Integration

Continious integration server er alveg clean, þannig sjáum við skýrar dependency. Integration server er server eða virtual umhverfi sem er er sett upp í þeim eina tilgangi að vinna í ákveðnu umhverfi.

Version Control = Revision Control = Source Control

Kerfi voru með lása, minna um það í dag. Hægt að track'a breytingar aftur í tímann.

Trunk

  • The unique line of development that is not a branch(sometimes also called a Baseline or mainline)
  • Sonja segir a' þetta sé aðaldraslið, what gonna do with all that junk, all that junk inside your trunk

Branch

  • Þá er unnið á annari útgáfu af dótinu, hægt að sameina seinna.
  • Semí sandbox

Tag

  • Er svona snapshot eða eð. Getur séð hver staðan var á skrám á einhverjum tímapunkti
  • Tagg er merkt eða skýrð sérstaklega.

Allskonar annað

  • Head - nýjasta commitið
  • Revision - er ein keðja af breytingum
  • Merge - oftast eru ekki breytingar á sama stað. Conflict eru ekki hægt að automate'a
  • Revert - Lagar manns eigið sandbox aftur að þeim punkti sem seinasta uppfærlsa var gerð
  • Rollback - Hættir við commit eða bakkar um commit.. veit ekki hvort

Key advantages

  • Allir eru með sitt sandbox
  • Þetta virkar off the line
  • Hratt
  • Smíðað til að deila breitingum
  • Branching og Merging er þægilegt
  • Less management

Disadvantage

  • Það er í rauninni aldrei til seinasta útgáfa

  • Er talin hluti af source code

Continious Integration

Passa að liggja ekki á gömlum kóða, byrja daginn á því. Gerir breitingar, pullar, ef kóðinn þinn keyrir enþá að þá pusharu.

Það er mjög erfitt að skrifa test fyrir klasa sem var ekki skrifaður með test í huga. Láta forritið stækka með þér, eða build skriptuna eða eð. Hraðinn á buildunum skiptir máli. Mar er með build á sinni vél og á integration vélinni. Hæg test er oftast ástæðan fyrir hægum buildum. Ekki láta unit test sækja gögn af harðadisk eða úr gagnasafni. Unit test eiga að keyra í minni.

4 Atriði sem verða að vera til staðar fyrir Integration Process

  1. Tenging við version control repo
  2. Build scripta
  3. Einhvers konar feedback mekanismi
  4. A process for integrating the source code changes

Manual Integration - Hefur integration token sem hver sækir sem ætlar að fara intergrate'a. Á meðan þú ert með tokenið, sem getur verið brúða eða whatever, oftast hefur fólk bara 15-30 mínútur til að integrate'a.

Continious Integration - Er server sem er stöðugt að athuga nokkuð að það séu komnar einhverjar breitingar. Ef þær eru komnar að þá keyrir hann build scripturnar og gefur svo grænt ef það er allt í lagi. Peoples eru búin að þróa allskonar visual feedback, eins og lava lampa og traffík ljós.

Verðmæti CI

  • Minnkuð áhætta
  • Bombar út deployable shitti hvenar sem er
  • Hannes skrifar ekki kóða í kaós

Good CI Practices

  • Commitaðu kóða oft
  • Ekki commita brotnum kóða
  • Lagaðu brotin build STRAXZ
  • Skrifaðu automated developer test, tryggðu að kóðinn þinn keyri rétt
  • Öll test og inspections verða að pass
  • Gerðu private build, pullaðu öllu og tékkaðu nokkuð það keyri ekki örugglega locally
  • Passaðu að fá ekki brotin kóða, þú tapar tíma á að pulla brotnu shitti

Test Double

SUT = System Under Test

DOC = Dependent-on Component

Test Doubles

  • Dummy objects eru sendir fram og tilbaka en aldrei í raun notaðir. Hafa vanalega bara það hlutverk að filla parametra.
  • Fake hafa í raun virkandi virkni, en stytta sér einhverjar leiðir og eru ekki æskilegar í production(GagnabankiIMinni t.d.)
  • Stubs skila "canned" svörum til fallakalla sem eru gerð í testi, en gera ekkert annað en bara það sem þau eru forrituð fyrir í prófuninni.
  • Spies eru stubs sem geyma upplýsingar um hvernig var kallað á þá. Ein tegund af spie er t.d. fyrir email þjónustu sem vistar hversu marga maila hann hefur sennt.
  • Mocks eru fyrirfram forritaðir(ég veit) með væntingum um köllin sem þeir gera ráð fyrir. Þeir geta kasstað exception ef þeir fá kall sem þeir eru ekki að búast við, og þeir eru athugaðir nokkuð þeir hafa ekki örugglega fengið öll köll sem þeir bjuggust við að fá.
